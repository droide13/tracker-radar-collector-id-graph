## 1. The Full CDP Network Domain Event Lifecycle

### The Two Parallel Streams

Chrome's network stack runs inside the **browser process** (not the renderer). The renderer asks the browser for resources; the browser fetches them and streams bytes back. CDP observes this from the browser side.

This architectural split is why there are two event streams:

- One carries what the renderer **intended** to send
- One carries what actually went over the wire, after the browser process applied its cookie policies

References: [`Network.requestWillBeSent`][rws] ¬∑ [`Network.requestWillBeSentExtraInfo`][rwsei] ¬∑ [Chrome `NetworkHandler` source][nh-src]

[rws]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-requestWillBeSent
[rwsei]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-requestWillBeSentExtraInfo
[nh-src]: https://source.chromium.org/chromium/chromium/src/+/main:content/browser/devtools/protocol/network_handler.cc

### Event Order for a Single HTTP/1.1 Request

```
Network.requestWillBeSent
  ‚Üì  (socket may be allocated here or after DNS)
Network.requestWillBeSentExtraInfo     ‚Üê parallel stream, may arrive before or after ‚Üë
  ‚Üì
  [DNS resolution]
  [TCP connect / TLS handshake]
  ‚Üì
Network.responseReceived
  ‚Üì
Network.responseReceivedExtraInfo      ‚Üê parallel stream
  ‚Üì
Network.dataReceived  (0..N times)
  ‚Üì
Network.loadingFinished  OR  Network.loadingFailed
```

For HTTP/2 and HTTP/3 (QUIC), there is no per-request TCP handshake because the connection is multiplexed; socket events do not appear on a per-request basis.

### Each Event in Detail

#### `Network.requestWillBeSent`

Fired by the renderer the moment it decides to make a request.

| Field | Description |
|---|---|
| `requestId` | Globally unique string for this request within the page session. Format is typically `"<frameId>.<sequenceNumber>"` ‚Äî treat as opaque. |
| `loaderId` | Identifies the document load this request belongs to. Associates requests with navigation. |
| `documentURL` | URL of the document initiating the request. |
| `request` | `{url, method, headers, postData, initialPriority, referrerPolicy}` ‚Äî headers as the renderer sees them, **without** HttpOnly/Secure cookies. |
| `timestamp` / `wallTime` | Monotonic and wall-clock timestamps. |
| `initiator` | `{type: "script"|"parser"|"other", url, lineNumber}` ‚Äî builds initiator chains. |
| `redirectResponse` | If this event represents a redirect, contains the **previous** response that triggered it. One `requestId` gets multiple `requestWillBeSent` events per redirect leg. |
| `type` | Resource type: `"Document"`, `"Script"`, `"XHR"`, `"Fetch"`, `"Image"`, etc. |
| `frameId` | Which frame initiated this request. |

Reference: [`Network.requestWillBeSent`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-requestWillBeSent)

#### `Network.requestWillBeSentExtraInfo`

Fired by the **browser process** when it constructs the actual wire request. Same `requestId`.

| Field | Description |
|---|---|
| `headers` | **Actual** headers sent over the wire, including `Cookie`. HttpOnly cookies are visible here. |
| `associatedCookies` | Array of `AssociatedCookie` objects with cookie value and `blockedReasons` (`SameSiteLax`, `SecureOnly`, `NotOnPath`, etc.) |
| `connectTiming` | When the socket was connected. |

> **‚öÝÔ∏è Ordering caveat:** `requestWillBeSentExtraInfo` can arrive **before** `requestWillBeSent` in CDP delivery order on fast connections. Correlate by `requestId` and handle either arrival order.

Reference: [`Network.requestWillBeSentExtraInfo`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-requestWillBeSentExtraInfo)

#### `Network.responseReceived`

Fired when response headers are fully received.

| Field | Description |
|---|---|
| `response.status` | HTTP status code. |
| `response.headers` | Renderer-visible response headers. |
| `response.mimeType` | MIME type. |
| `response.protocol` | `"h2"`, `"h3"`, `"http/1.1"`, etc. |
| `response.timing` | `{requestTime, proxyStart/End, dnsStart/End, connectStart/End, sslStart/End, sendStart/End, receiveHeadersEnd}` ‚Äî all in ms relative to `requestTime`. Raw data for HAR `timings`. |
| `response.fromDiskCache` | `true` if served from disk cache. |
| `response.fromServiceWorker` | `true` if intercepted by a service worker. |

Reference: [`Network.responseReceived`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-responseReceived)

#### `Network.responseReceivedExtraInfo`

Same `requestId`. Contains the actual wire response.

| Field | Description |
|---|---|
| `headers` | Actual wire response headers, including `Set-Cookie` with full HttpOnly cookie values. **Only place HttpOnly cookies are observable in CDP.** |
| `blockedCookies` | Cookies received but not stored, with `blockedReasons`. |
| `headersText` | Raw header text for exact HAR `headersSize` calculation. |
| `statusCode` | Actual HTTP status code. |

Reference: [`Network.responseReceivedExtraInfo`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-responseReceivedExtraInfo)

#### `Network.dataReceived`

Fires for each chunk of response body delivered to the renderer.

| Field | Description |
|---|---|
| `dataLength` | Bytes delivered to the renderer. |
| `encodedDataLength` | Compressed wire bytes. Sum across all `dataReceived` events for total wire size. |

Reference: [`Network.dataReceived`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-dataReceived)

#### `Network.loadingFinished`

Fires when the response body is complete. After this event, `getResponseBody` will succeed ‚Äî until it doesn't (see [¬ß5](#5-response-body-buffering)).

| Field | Description |
|---|---|
| `encodedDataLength` | Total compressed wire bytes. Should match sum of `dataReceived.encodedDataLength`. |

Reference: [`Network.loadingFinished`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-loadingFinished)

#### `Network.loadingFailed`

Fires instead of `loadingFinished` on request failure.

| Field | Description |
|---|---|
| `errorText` | e.g. `"net::ERR_CONNECTION_REFUSED"` |
| `canceled` | `true` if user navigated away. |
| `blockedReason` | For content-blocking. |
| `corsErrorStatus` | Details for CORS failures. |

Reference: [`Network.loadingFailed`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-loadingFailed)

### Why Two Streams for Cookies

The separation is a **security boundary**, not a debugging convenience. Chrome's multi-process architecture puts the renderer in an isolated process with no access to the cookie store. The browser process holds all cookies.

When the renderer asks for a resource, it sends a `URLRequest`; the browser process attaches cookies to the actual `HttpRequestHeaders` just before writing to the socket.

- `requestWillBeSent` is instrumented at the renderer's `ResourceRequest` creation point
- `requestWillBeSentExtraInfo` is instrumented at the `URLRequest` level in the browser process, after [`HttpCookieInclusionStatus`](https://source.chromium.org/chromium/chromium/src/+/main:net/cookies/cookie_inclusion_status.h) evaluation

The two events travel through separate IPC pipes and arrive on different timing schedules.

## 2. Network Idle and Almost-Idle

### Chrome's Internal Definition

Chrome tracks "in-flight" requests as the count of active [`URLLoader`](https://source.chromium.org/chromium/chromium/src/+/main:services/network/public/mojom/url_loader.mojom) connections for a given page:

| Puppeteer constant | State | In-flight threshold | Debounce |
|---|---|---|---|
| [`networkidle0`][ni0] | Network idle | 0 requests | 500ms |
| [`networkidle2`][ni2] | Network almost idle | ‚â§2 requests | 500ms |

The 500ms is Puppeteer's polling debounce, **not** a Chrome-internal concept. Chrome doesn't expose idle state directly over CDP; Puppeteer synthesizes it by tracking `requestWillBeSent` (increment) and `loadingFinished`/`loadingFailed` (decrement).

[ni0]: https://pptr.dev/api/puppeteer.puppeteerlifecycleevent
[ni2]: https://pptr.dev/api/puppeteer.puppeteerlifecycleevent

### Puppeteer's Implementation

[`waitForNavigation({waitUntil: 'networkidle0'})`](https://pptr.dev/api/puppeteer.page.waitfornavigation) maintains an in-flight counter. When the counter hits the threshold, a 500ms timer starts. If the counter rises above the threshold before the timer fires, the timer resets. When the timer fires, navigation is considered complete.

### Failure Modes

| Failure Mode | Explanation |
|---|---|
| **Long-polling / persistent XHR** | An open XHR waiting for server push holds the counter at ‚â•1. `networkidle0` never fires. |
| **SSE ([`EventSource`][sse])** | Persistent connection. `networkidle2` handles SSE + 1 other; many sites exceed this. |
| **[WebSockets][ws]** | WebSocket connections are **not counted** in the in-flight counter. `webSocketCreated` fires but does not block idle. |
| **SPA deferred fetches** | Data fetching in a `setTimeout(..., 0)` after initial load ‚Äî network goes idle, Puppeteer resolves, second wave of requests is missed entirely. |
| **Service worker precaching** | SW installing and making precache requests goes through the SW session's network stack, invisible to the page session's idle counter. |
| **QUIC 0-RTT** | `requestWillBeSent` may fire before session is ready to track it, causing counter drift. |

[sse]: https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events
[ws]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-webSocketCreated

> **Practical recommendation:** Use `networkidle2` with a hard timeout rather than `networkidle0`. Also independently track `Page.loadEventFired` as a floor ‚Äî do not declare done until load has fired.

## 3. Target Types and Their Network Isolation

### Why Each Target Has Its Own CDP Session

A CDP session is a 1:1 mapping to a Chrome DevTools session attached to a specific [`Target`](https://chromedevtools.github.io/devtools-protocol/tot/Target/). A Target in Chrome corresponds to a [`RenderFrameHost`][rfh] (for pages/iframes) or a worker context. Each target runs in an independent V8 context with its own [`URLLoaderFactory`][ulf] instance scoped to its origin and storage partition.

CDP sessions are multiplexed over a single WebSocket using `sessionId` as a routing key.

When you call `Network.enable` on session A, Chrome attaches network instrumentation hooks to the `URLLoaderClient` for target A's process. Session B gets no notifications because B's `URLLoaderClient` is a different object in a different process.

[rfh]: https://source.chromium.org/chromium/chromium/src/+/main:content/browser/renderer_host/render_frame_host_impl.h
[ulf]: https://source.chromium.org/chromium/chromium/src/+/main:services/network/public/mojom/url_loader_factory.mojom

### Cross-Process Iframe Network Isolation

When Chrome places an iframe in its own renderer process, the iframe's network events fire on **that process's** DevTools session. There is no "bubbling" of network events to the parent page session because the events are emitted by the `URLLoaderClient` in the child process, which has no knowledge of the parent's DevTools session.

If the iframe is same-process (same-origin, same renderer), its requests **do** appear on the parent's CDP session with the iframe's `frameId`. This is why naively attaching only to the top-level page session produces incomplete HAR files for cross-origin heavy pages.

### Site Isolation: When Does Chrome Use a New Process?

Chrome's [Site Isolation][si] policy (enabled by default on desktop since Chrome 67) puts each *site* (scheme + eTLD+1) in its own renderer process:

| Scenario | Process assignment |
|---|---|
| Cross-site iframe (e.g. `evil.com` inside `good.com`) | Always gets its own renderer process on desktop Chrome |
| Same-site iframe (e.g. `sub.good.com` inside `good.com`) | Shares parent renderer ‚Äî network events share parent's CDP session |
| [`Origin-Agent-Cluster`][oac] + COOP/COEP | Origin-level isolation; `sub.good.com` gets its own process even if eTLD+1 matches |
| Process cap hit | Chrome may coalesce sites into shared renderers ‚Äî cross-site iframes may share a session non-deterministically |
| `<iframe sandbox>` without `allow-same-origin` | Unique origin, always gets its own process |

[oac]: https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Origin-Agent-Cluster

### Why `Network.enable` Must Be Called Per Session

[`Network.enable`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#method-enable) is a command to Chrome's `DevToolsSession` object, scoped to a single target. It sets a flag in that session's `NetworkHandler` instance. There is no global `Network.enable` for all sessions. Session B's `NetworkHandler` is a separate object, entirely unaffected by enabling on session A.

## 4. Page Lifecycle Events

### The HTML Parsing Pipeline

```
Navigation start
  ‚Üì
Response headers received ‚Üí HTML parser starts
  ‚Üì
Parser discovers <script>, <link>, <img> ‚Üí subresource requests fire
  ‚Üì
Parser blocked by sync scripts (executes them)
  ‚Üì
Parser reaches </html> / deferred scripts complete
  ‚Üì
DOMContentLoaded fires  (parser done, sync + defer complete; images may still load)
  ‚Üì
All subresources (images, iframes, etc.) complete
  ‚Üì
load event fires on window
```

References: [MDN ‚Äî `DOMContentLoaded`](https://developer.mozilla.org/en-US/docs/Web/API/Document/DOMContentLoaded_event) ¬∑ [MDN ‚Äî `load` event](https://developer.mozilla.org/en-US/docs/Web/API/Window/load_event)

### CDP Events and Their Mapping

| CDP Event | Trigger | Notes |
|---|---|---|
| [`Page.frameStartedLoading`][pfs] | Frame begins a load, before response headers | Brackets the full load lifecycle (open) |
| [`Page.frameNavigated`][pfn] | Frame commits a navigation (new URL is now current) | Not the same as navigation start ‚Äî may be delayed |
| [`Page.domContentEventFired`][pdcef] | Maps to `DOMContentLoaded` | After parser + deferred scripts done, before `load` |
| [`Page.loadEventFired`][plef] | Maps to `window.load` | All subresources complete. Reliable "done" signal for crawlers |
| [`Page.frameStoppedLoading`][pfsl] | Frame load fully complete (after `load` event) | Brackets the full load lifecycle (close) |
| [`Page.frameRequestedNavigation`][pfrn] | Script calls `location.href` or form submits | Before navigation starts |
| [`Page.navigatedWithinDocument`][pnwd] | Hash change or History API (`pushState`/`replaceState`) | No network request; URL changes only |

## 5. Response Body Buffering

### Why Chrome Buffers Response Bodies

CDP's [`Network.getResponseBody`][grb] is implemented by Chrome's [`DevToolsNetworkInterceptor`][dni], which maintains an in-memory cache of response bodies keyed by `requestId`. When `Network.enable` is active, Chrome's `NetworkHandler` intercepts all response data before it's discarded and stores it in this cache in the browser process.

[grb]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#method-getResponseBody
[dni]: https://source.chromium.org/chromium/chromium/src/+/main:content/browser/devtools/devtools_network_interceptor.h

### Buffer Limits

[`Network.enable`](https://chromedevtools.github.io/devtools-protocol/tot/Network/#method-enable) accepts two optional size limits:

| Parameter | Default | Behavior when exceeded |
|---|---|---|
| `maxTotalBufferSize` | 10 MB | Older entries evicted (LRU-ish) from the total cache |
| `maxResourceBufferSize` | 5 MB | Single response truncated in the buffer (full response still delivered to page) |

When a response body is evicted or truncated, `getResponseBody` may still return a partial result or an error depending on how much was buffered.

### When `getResponseBody` Fails

| Cause | Explanation |
|---|---|
| Request completed before `Network.enable` | Chrome only buffers bodies for requests observed after enabling. |
| Buffer evicted | Large page with many large resources ‚Äî early responses evicted from the 10 MB total buffer. |
| No body | 204 No Content, 304 Not Modified, HEAD requests. |
| Streaming response | SSE, indefinite chunked transfer ‚Äî `loadingFinished` never fires. |
| Wrong session | `requestId` used on a different session than the one that received `loadingFinished`. See below. |

### Why `getResponseBody` Must Use the Same Session

The response body cache is maintained **per CDP session**, not globally. When session A's `NetworkHandler` buffers a response, it's stored in a map owned by session A's handler. Session B has a completely separate map.

If `loadingFinished` fires on session A (e.g. for a cross-process iframe), calling `getResponseBody` with that `requestId` on session B will return `"No resource with given identifier found"`.

> **This is why your `loadingFinished` listener must close over its own session reference and call `getResponseBody` on that session, not on a shared/central session.**

## 6. Target Attachment Lifecycle and `waitForDebuggerOnStart`

### The Full Attachment Flow

When [`Target.setAutoAttach`][tsa] is called with `{autoAttach: true, waitForDebuggerOnStart: true, flatten: true}`, Chrome will:

1. Pause the new target's execution before any JavaScript runs
2. Emit [`Target.attachedToTarget`][tat] carrying the new `sessionId` and `TargetInfo`
3. Wait for [`Runtime.runIfWaitingForDebugger`][riwd] to be called on the new session

```
Target created
  ‚Üì
Target.attachedToTarget fires  ‚Üí  you create a CDPSession
  ‚Üì
[Target paused ‚Äî no JS runs, no network requests]
  ‚Üì
You call: Network.enable, Target.setAutoAttach (on child), other domain enables
  ‚Üì
You call: Runtime.runIfWaitingForDebugger
  ‚Üì
Target resumes execution ‚Üí first request fires (you catch it)
```

[tsa]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#method-setAutoAttach
[tat]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-attachedToTarget
[riwd]: https://chromedevtools.github.io/devtools-protocol/tot/Runtime/#method-runIfWaitingForDebugger

### Why `waitForDebuggerOnStart: true` Is Critical

Without it, there is a race: Chrome emits `attachedToTarget`, then immediately starts executing the target. By the time your JavaScript processes the event and calls `Network.enable`, the target may have already made its first (or many) requests. You'll have incomplete instrumentation.

With `waitForDebuggerOnStart: true`, target execution is paused at the very start of the V8 context ‚Äî before any scripts run, before any network requests are made. All domain-enables must happen **before** calling `runIfWaitingForDebugger`. This is the only window for guaranteed-complete instrumentation.

### What Happens If You Enable After `runIfWaitingForDebugger`

For a fast-loading page, the renderer may fetch the main document, several scripts, and stylesheets before your `Network.enable` call completes its round-trip. You'll see `responseReceived` and `loadingFinished` events for requests whose `requestWillBeSent` you never received ‚Äî incomplete HAR entries. On localhost you may get lucky. On production, you will consistently miss the first wave of requests.

### `setAutoAttach` and Its Propagation

`Target.setAutoAttach` sets auto-attach for the session you call it on only. To get auto-attach for grandchild targets (iframes within iframes, workers spawned by workers), you must call `Target.setAutoAttach` again on each newly attached session.

> **Pattern:** In every `attachedToTarget` handler, after enabling domains, call `Target.setAutoAttach` on the new session so that its children are also auto-attached.

## 7. Race Conditions in Multi-Session CDP Instrumentation

### Why Chrome Attaches the Same Target Twice

Chrome can produce duplicate attachments in several scenarios:

**Service workers:** A service worker registered in a previous page load may already have an existing DevTools session. When your crawler session calls `setAutoAttach`, Chrome processes existing targets and attaches them. If the service worker was already attached, Chrome creates a second session for the same underlying target.

**`setAutoAttach` timing:** If called while a target is being created (between `targetCreated` and Chrome finishing session setup), Chrome may attach it once via auto-attach and once via the explicit list of existing targets. Both calls produce different `sessionId`s.

**Which session survives:** Chrome doesn't kill the old session. You receive `attachedToTarget` for both. The second session is canonical going forward. The first session's underlying `CDPClient` is still technically connected (WebSocket is open), but Chrome may have torn down state for that target under the first session. Commands to the first session typically produce `"Target already exists: old session X, new Y"`, `"Target closed"`, or `"Session closed"` errors.

### When a CDPSession Becomes Closed

A [`CDPSession`](https://pptr.dev/api/puppeteer.cdpsession) becomes closed when either:
- Chrome sends [`Target.detachedFromTarget`](https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-detachedFromTarget) for its `sessionId`, or
- The underlying WebSocket connection closes

Puppeteer marks the session closed **synchronously** when it processes these events, but event processing is asynchronous (next microtask tick after the WebSocket message arrives).

`session.connection()` returns `null` after the session is marked closed. This is a reliable signal *after* the detach event has been processed. But there is a window where a session is physically closed and `connection()` still returns non-null.

> **Do not rely on `connection() !== null` as a pre-flight check.** Wrap all CDP commands in try/catch and handle `TargetCloseError` / `"Session closed"` errors gracefully.

### Why `Promise.allSettled` Is Always Correct for Multi-Session Work

| | [`Promise.all`][pall] | [`Promise.allSettled`][pas] |
|---|---|---|
| If one rejects | Rejects immediately (fail-fast) | Waits for all to settle |
| Return value | Array of resolved values | Array of `{status, value/reason}` |
| Use when | All must succeed; failure is fatal | Partial failure is expected and acceptable |
| In CDP multi-session work | Almost never correct | Always correct for domain enables |

[pall]: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/all
[pas]: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/allSettled

```javascript
const results = await Promise.allSettled(
  sessions.map(s => s.send('Network.enable', {}))
);
results.forEach((r, i) => {
  if (r.status === 'rejected') {
    this._log?.(`Session ${i} Network.enable failed: ${r.reason.message}`);
  }
});
```

### Other CDP Commands Vulnerable to the Same Dead-Session Race

| Command | Vulnerability |
|---|---|
| `Runtime.runIfWaitingForDebugger` | Most critical ‚Äî if target detaches while paused, will never resolve without a timeout |
| `Page.enable`, `Log.enable`, `DOM.enable` | All domain enables |
| `Runtime.evaluate` | Extremely vulnerable on worker/service-worker targets that may be shut down |
| `Target.setAutoAttach` on a child session | If the child is already dead |
| `Network.getResponseBody` | If target closed between `loadingFinished` and your call (common with ephemeral service workers) |

### `targetCreated` / `targetDestroyed` / `targetInfoChanged` vs Session Validity

| Event | Session validity implication |
|---|---|
| [`Target.targetCreated`][ttc] | Target exists; you may not be attached yet |
| [`Target.targetDestroyed`][ttd] | Target is gone. All attached sessions are now dead. Remove all state for those sessions immediately. |
| [`Target.targetInfoChanged`][ttic] | Metadata changed (URL, title). Session is still valid. |
| [`Target.detachedFromTarget`][tdft] | **Your specific session handle is dead**, regardless of whether the underlying target still exists. |

[ttc]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-targetCreated
[ttd]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-targetDestroyed
[ttic]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-targetInfoChanged
[tdft]: https://chromedevtools.github.io/devtools-protocol/tot/Target/#event-detachedFromTarget

`targetDestroyed` ‚Üí `detachedFromTarget` is the expected sequence when Chrome closes a target and auto-detaches your session. But `detachedFromTarget` can also fire without `targetDestroyed` (explicit detach, session reuse).

### The Correct Defensive Pattern for Unknown/Ephemeral Sessions

```javascript
/**
 * @param {import('puppeteer-core').CDPSession} session
 * @param {import('devtools-protocol/types/protocol').Protocol.Target.TargetInfo} targetInfo
 */
async addTarget(session, targetInfo) {
  // 1. Register cleanup for when this session dies
  session.on('Target.detachedFromTarget', () => {
    this._cleanupSession(session);
  });

  // 2. Enable all required domains ‚Äî don't let one failure kill others
  await Promise.allSettled([
    session.send('Network.enable', {
      maxTotalBufferSize: 20 * 1024 * 1024,
      maxResourceBufferSize: 5 * 1024 * 1024,
    }),
    session.send('Target.setAutoAttach', {
      autoAttach: true,
      waitForDebuggerOnStart: true,
      flatten: true,
    }),
  ]);

  // 3. Register event listeners ‚Äî closures correctly capture this session
  session.on('Network.requestWillBeSent', event => {
    this._handleRequest(session, event);
  });

  session.on('Network.loadingFinished', async event => {
    await this._fetchBody(session, event); // session captured in closure
  });

  // 4. Only now, release the debugger pause (if applicable)
  await session.send('Runtime.runIfWaitingForDebugger').catch(() => {
    // Not paused or already resumed ‚Äî that's fine
  });
}

async _fetchBody(session, event) {
  try {
    const body = await session.send('Network.getResponseBody', {
      requestId: event.requestId,
    });
    this._storeBody(event.requestId, body);
  } catch (e) {
    if (!e.message.includes('No resource') && !e.message.includes('Session closed')) {
      throw e;
    }
    // Expected: body evicted, session closed, no body for this request type
  }
}
```

## 8. Practical Failure Modes for HAR Capture

### Events Arriving Out of Order

CDP events are delivered over a WebSocket, which is ordered per-connection. However, events from **different sessions** arrive on the same WebSocket interleaved non-deterministically. `requestWillBeSent` from session A and `requestWillBeSentExtraInfo` from session A may be separated by 50 events from session B.

Your collector must buffer events by `requestId` and assemble complete request entries lazily. Also, `requestWillBeSentExtraInfo` can arrive before `requestWillBeSent` for the same `requestId` ‚Äî handle both orderings.

Reference: [HAR 1.2 Spec](http://www.softwareishard.com/blog/har-12-spec/) ¬∑ [`chrome-har`](https://github.com/sitespeedio/chrome-har)

### `requestId` Uniqueness Guarantees and Their Limits

- `requestId` is unique **within a single CDP session**
- It is **not** unique **across sessions** ‚Äî two different sessions can produce the same `requestId` string for different requests
- `requestId` is **reused within a session for redirect chains** ‚Äî each redirect leg uses the same `requestId` with a new `requestWillBeSent` event

> **When merging events from multiple sessions into a single HAR, namespace requestIds:** `"${sessionId}::${requestId}"`

### Cached Responses

| Cache type | Events fired | `getResponseBody` |
|---|---|---|
| Memory cache | `requestWillBeSent` + [`Network.requestServedFromCache`][nrsc] + `loadingFinished` | Fails ‚Äî no body buffered |
| Disk cache | Full sequence with `response.fromDiskCache === true` | Succeeds normally |

[nrsc]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-requestServedFromCache

### Redirects

Each redirect leg fires `requestWillBeSent`. The second (and subsequent) `requestWillBeSent` events carry a `redirectResponse` field containing the 301/302 response. The `requestId` stays the same across all legs.

> **If you use a `Map<requestId, entry>` and overwrite on each `requestWillBeSent`, you'll lose all but the final leg.** `chrome-har` creates one HAR entry per `requestWillBeSent` event with a `redirectURL` in the response.

### Preflight CORS Requests

[CORS preflight](https://developer.mozilla.org/en-US/docs/Glossary/Preflight_request) (`OPTIONS`) requests fire as **independent events** with their own `requestId`. They are not linked to the actual request at the CDP level. Your HAR will have two entries with no explicit link. `chrome-har` does not currently merge them.

### Streaming Responses (SSE, chunked)

For SSE and long-running chunked responses, `loadingFinished` never fires until the connection closes. Options:

- Set a timeout and call `getResponseBody` after the timeout regardless
- Detect SSE by content-type (`text/event-stream`) on `responseReceived` and handle separately
- Accept that SSE body capture is inherently incomplete

Reference: [MDN ‚Äî Server-Sent Events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events)

### Service Worker Fetch Interception

When a service worker intercepts a fetch:
- The page session sees `requestWillBeSent` normally
- `responseReceived` may carry `fromServiceWorker: true`
- If the SW responds from its own cache, **no network events fire at all** in the network stack
- The SW's CDP session will show `fetch` events in its `Network` domain (its own outgoing requests, not the interception itself)

### QUIC/HTTP3

QUIC requests appear normally in the CDP Network domain with `response.protocol === "h3"` (or `"h3-29"`, etc.). On 0-RTT repeat visits, the connection is established before any DNS/connect events ‚Äî `timing.connectStart` may be `-1`.

> **Always clamp timing values to 0** to prevent negative HAR timings.

Reference: [Chrome HTTP3 overview](https://developer.chrome.com/en/docs/web-platform/http3/)

### Responses with No Body

| Response type | `getResponseBody` behavior |
|---|---|
| 304 Not Modified | Fails ‚Äî no body |
| 204 No Content | Fails ‚Äî no body |
| HEAD requests | Fails ‚Äî no body by definition |
| 101 WebSocket Upgrade | Body is WebSocket frames ‚Äî use [`Network.webSocketFrameReceived`][wsfr] |

[wsfr]: https://chromedevtools.github.io/devtools-protocol/tot/Network/#event-webSocketFrameReceived

HAR entries for these should have `bodySize: 0`.

## 9. Advanced JavaScript and JSDoc Patterns

### `async addTarget()` Awaited by the Crawler

When the framework does `await collector.addTarget(session, targetInfo)`, it waits for the returned promise to settle. If `addTarget` is declared `async`, it automatically returns a `Promise` that resolves when the function body completes.

**Failure mode ‚Äî forgot `async`, used `.then()` without returning:**

```javascript
// WRONG ‚Äî returns undefined; framework proceeds immediately
addTarget(session, targetInfo) {
  session.send('Network.enable', {}).then(() => {
    session.on('Network.loadingFinished', ...); // registered too late
  });
  // implicit return undefined
}
```

The framework's `await` on `undefined` resolves immediately. `Network.enable` hasn't completed; your event listeners aren't registered. On fast-loading pages, you miss everything.

**Correct:**

```javascript
async addTarget(session, targetInfo) {
  await session.send('Network.enable', {});
  session.on('Network.loadingFinished', this._handleFinished.bind(this, session));
}
```

### Closures Over `session` in Loops

```javascript
for (const session of sessions) {
  session.on('Network.loadingFinished', async (event) => {
    const body = await session.send('Network.getResponseBody', {
      requestId: event.requestId,
    });
  });
}
```

Each iteration of a `for...of` loop creates a new block scope. `session` is `const`-scoped to each iteration. The closure captures the specific `session` from that iteration. This is correct.

**This differs from the classic `var` closure bug:**

```javascript
// WRONG ‚Äî classic var bug
for (var i = 0; i < sessions.length; i++) {
  sessions[i].on('Network.loadingFinished', async (event) => {
    // i is shared ‚Äî by the time this fires, i === sessions.length
    const body = await sessions[i].send(...); // sessions[i] is undefined
  });
}
```

`var` is function-scoped ‚Äî all closures share the same `i`, which is `sessions.length` by the time any listener fires. `const`/`let` in `for...of` and `for (let ...)` create a new binding per iteration.

Reference: [MDN ‚Äî Closures](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures) ¬∑ [MDN ‚Äî `var` hoisting](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/var#hoisting)

### `Set<Promise<void>>` for In-Flight Tracking

```javascript
/** @type {Set<Promise<void>>} */
this._inFlight = new Set();

async _fetchBody(session, event) {
  const promise = (async () => {
    try {
      const body = await session.send('Network.getResponseBody', {
        requestId: event.requestId,
      });
      this._store(event.requestId, body);
    } catch {}
  })();

  this._inFlight.add(promise);
  promise.finally(() => this._inFlight.delete(promise));
}

async getData() {
  await Promise.allSettled([...this._inFlight]);
  return this._buildHAR();
}
```

The pattern: add to the Set immediately, remove in `.finally()`. `.finally()` runs whether the promise resolves or rejects ‚Äî the Set never leaks stale promises.

The IIFE `(async () => { ... })()` creates the promise immediately (before any `await` inside suspends), so `this._inFlight.add(promise)` and `promise.finally(...)` both run synchronously in the same tick. This guarantees the Set is populated before the event loop tick where `await session.send(...)` suspends.

### `Promise.allSettled` vs `Promise.all`

See the [table in ¬ß7](#why-promiseallsettled-is-always-correct-for-multi-session-work).

Use `Promise.all` when you need all results and a failure should abort ‚Äî e.g. parallel `getResponseBody` calls where you need every body. Use `Promise.allSettled` when firing CDP commands to N sessions where some sessions may be dead.

### JSDoc Cross-Package Type Imports

```javascript
/**
 * @param {import('puppeteer-core').CDPSession} session
 * @param {import('devtools-protocol/types/protocol').Protocol.Target.TargetInfo} targetInfo

```

`import('package').Type` in JSDoc is a TypeScript-compatible type import that works without a compilation step. The TypeScript language server resolves these types by:

1. Following Node's `require.resolve` logic to find the package's `package.json`
2. Reading `types` or `typings` field to find the `.d.ts` file
3. Resolving the named export from that file

For `devtools-protocol`, `devtools-protocol/types/protocol` points to a `.d.ts` file exporting a `Protocol` namespace. `Protocol.Target.TargetInfo` is the exact shape of the `targetInfo` object Chrome sends.

### `@typedef` Placement Rules

```javascript
/**
 * @typedef {Object} HAREntry
 * @property {string} requestId
 * @property {number} startedDateTime
 */

/**
 * @param {HAREntry} entry  ‚Üê works because typedef is above
 */
function processEntry(entry) { ... }
```

JSDoc `@typedef` declarations are **not hoisted**. The TypeScript language server processes JSDoc comments top-to-bottom. If `@typedef HAREntry` appears *after* the function that references `@param {HAREntry}`, the type checker may report `HAREntry` as unknown at the point of use.

This differs from JavaScript's `function` hoisting (a JS engine behavior) ‚Äî JSDoc is parsed by the type checker in a linear pass.

> **Place `@typedef` blocks before their first use**, ideally at the top of the file or in a dedicated `types.js` file.

Reference: [JSDoc `@typedef`](https://jsdoc.app/tags-typedef) ¬∑ [JSDoc `@param`](https://jsdoc.app/tags-param)

### The `?.()` Optional Call Operator

```javascript
this._log?.('Session attached', sessionId);
```

`?.()` is the [optional call operator](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/Optional_chaining). It calls the function only if the expression to its left is not `null` or `undefined`. If `this._log` is `undefined`, the expression evaluates to `undefined` without throwing `TypeError`.

This compiles (Babel/TypeScript) to approximately:

```javascript
(_this$_log = this._log) === null || _this$_log === void 0
  ? void 0
  : _this$_log.call(this, 'Session attached', sessionId);
```

| Pattern | Use when |
|---|---|
| `this._log?.(...)` | Logger is truly optional; no logger is a normal condition |
| `if (this._log) { this._log(...) }` | You need to distinguish `null` from `undefined`, or want explicit intent |
| `this._log(...)` directly | Logger is always required; fail loudly if missing |

Reference: [MDN ‚Äî Optional Chaining `?.`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/Optional_chaining)

## Putting It All Together: HAR Capture Architecture

Given the architecture (per-session `addTarget`, `chrome-har` event merging, DuckDuckGo Tracker Radar Collector fork), the correct data flow is:

```
init()
  ‚îî‚îÄ initialize: events[], bodyMap{}, inFlight Set

addTarget(session, targetInfo)  [awaited per session]
  ‚îú‚îÄ Promise.allSettled([Network.enable, Target.setAutoAttach])
  ‚îú‚îÄ on: requestWillBeSent       ‚Üí push to events[] (namespaced requestId)
  ‚îú‚îÄ on: requestWillBeSentExtraInfo ‚Üí push to events[]
  ‚îú‚îÄ on: responseReceived        ‚Üí push to events[]
  ‚îú‚îÄ on: loadingFinished         ‚Üí push event + launch getResponseBody (add to inFlight)
  ‚îú‚îÄ on: loadingFailed           ‚Üí push to events[]
  ‚îî‚îÄ Runtime.runIfWaitingForDebugger (catch all errors)

getData()
  ‚îú‚îÄ await Promise.allSettled([...inFlight])   ‚Üê wait for all pending body fetches
  ‚îî‚îÄ return chrome-har.harFromMessages(events, {bodies: bodyMap})
```

**The `requestId` namespacing** (`${sessionId}::${requestId}`) is the key to correct multi-session HAR merging without collision. Apply this prefix to `requestId` in each event object before pushing to the shared array, and ensure `getResponseBody` results are stored under the same namespaced key.

References: [`chrome-har`](https://github.com/sitespeedio/chrome-har) ¬∑ [HAR 1.2 Specification](http://www.softwareishard.com/blog/har-12-spec/) ¬∑ [DuckDuckGo Tracker Radar Collector](https://github.com/duckduckgo/tracker-radar-collector)
